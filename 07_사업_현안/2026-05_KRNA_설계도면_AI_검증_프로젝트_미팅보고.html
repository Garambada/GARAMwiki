Executive Summary

핵심 판단 — Key Judgements

판단 01
EREC 이탈 = 설계회사 풀 전체의 경고
EREC 이탈은 단일 파트너 이탈이 아니라 '설계회사 풀(pool) 전체가 우리에게 던지는 경고'로 해석해야 한다. 세종기술도 동일한 사유로 이탈할 가능성이 있어, 첫 미팅 이전에 'SME 부담 최소화 모델'을 명시적으로 제시해야 한다.
판단 02
과거 실패 = 우리의 결정적 포지셔닝 단서
고가 SW 1회 구입 → 업데이트 단절 → 폐기 패턴은 우리가 피해야 할 함정이자 차별화 메시지의 원천이다. WISE42의 차별점은 'AI 모델 납품'이 아니라 'Living System으로서의 AX 운영서비스'여야 한다.
판단 03
LLM 단독 불가 — Ontology가 모델보다 먼저다
기술적으로는 LLM 단독으로 풀 수 없으며, Ontology + RAG + Pricing Feed의 3-Layer Architecture가 필수다. 자재·공종·지역·속도등급·전압 등을 차원으로 갖는 분류체계를 먼저 설계해야 한다.

6대 도메인 난제 —
프로젝트의 본질적 리스크

아래 6개 난제는 EREC 측이 '성공 가능성 낮음' 판단의 근거로 제시한 항목이다. 각 난제는 솔루션 아키텍처·운영모델·계약전략에 1:1로 매핑되어 선반영되어야 한다.

🔴 난제 01
수량산출 업무 숙련에 신입→5년 소요

도면 해독 능력 + 철도전기 도메인 암묵지(tacit knowledge)가 핵심 자산. 문제는 '도면→수량' 변환이 아니라 '도면을 어떻게 읽어야 하는가에 대한 5년치 경험'을 LLM이 대체·증강할 수 있는가다.

→ 함의
Ontology + 준공도면 학습으로 5년치 암묵지를 디지털 자산화. 이것이 Living System의 핵심 가치.
🔴 난제 02
설계도면 오류는 준공 시 일괄 수정

설계변경이 잦아 준공 전 수정이 무의미하다는 업계 관행. 결과적으로 학습데이터로 직접 활용 가능한 '클린 도면'은 준공도면뿐이며, 설계 중 도면은 노이즈 포함 상태로 학습해야 함.

→ 함의
학습 데이터 거버넌스 설계 필수. 준공도면 우선 학습(클린 데이터셋), 설계 중 도면은 수정 이력과 함께 학습하여 변경 패턴까지 예측.
🔴 난제 03
수백 변종, SKU 1만+ 규모

동일 설비도 지역·궤도종류·운행속도·운행간격·전압·시공방식에 따라 수백 변종 발생. 철도전기 자재 마스터만 1만 개를 초과.

→ 함의
단순 OCR/룰베이스로는 절대 불가능. 분류체계(Ontology)가 모델보다 먼저 구축되어야 함. Layer 1이 가장 선행되어야 하는 이유.
🟡 난제 04
단가가 물가정보지·나라장터 미등재

철도전기 자재는 일반 전기공사 자재와 달라 공개 가격 DB 부재. 견적 기반으로만 단가 확보 가능하며, 6개월 단위로 가격이 변동.

→ 함의
단가 신선도(freshness)를 시스템이 자체 관리해야 함. 견적 미보유 자재 발생 시 RFQ 자동발송 이벤트로 정의. Layer 3의 핵심 기능.
🟡 난제 05
과거 유사 프로젝트의 실패 전례

설계사들이 고가의 산출 SW를 구입했으나, 업데이트가 단절되어 1회 사용 후 폐기됨.

→ 함의
본 프로젝트는 '제품(product)'이 아니라 '운영서비스(operating service)'로 포지셔닝해야 함. 구독형 + 자동 업데이트 + 정기 감사 = Living System.
🟠 난제 06
설계사 SME 인력의 상시 외근

설계사 인력 다수가 현장조사 등으로 상시 외근. 개발자의 실시간 질의응답을 받아주기 어려움.

→ 함의
'실시간 SME 의존' 협업 모델은 실패할 것. '비동기 수집·라벨링' 모델로 재설계 필요. 설계사에 묻는 게 아니라 설계사가 평소 만들어내는 결과물(견적요청서, 산출서, 도면 수정이력)을 수동으로 수집·라벨링·학습.
전략적 함의

4가지 구조적 리스크와 기회

🚨 설계회사 풀 전체의 경고 신호

EREC가 '리소스 부족'과 '성공 가능성 낮음' 두 가지를 동시에 사유로 든 점에 주목. 전자는 회사 사정이지만 후자는 업계 공통의 인식에 가깝다. 세종기술 역시 첫 검토에서 동일 결론에 도달할 가능성이 적지 않다.

→ '왜 이번에는 다른가'를 첫 미팅 이전에 자료로 입증해야 한다.

🎯 과거 실패 = 결정적 포지셔닝 단서

'고가 SW 1회 구입 → 업데이트 단절 → 폐기' 패턴은 우리가 피해야 할 함정이자, 동시에 차별화 메시지의 원천이다. 라이선스 판매가 아닌 구독형 운영서비스 모델, 자동 업데이트 파이프라인, 정기 정확도 감사로 Living System을 증명한다.

🏗️ Ontology가 모델보다 먼저다

1만+ SKU, 수백 변종, 6개월 단위 가격변동에서는 어떤 LLM도 단독으로 정답을 내지 못한다. 자재·공종·지역·속도등급·전압 등을 차원으로 갖는 분류체계(Ontology)를 먼저 설계하고, 그 위에 RAG와 Pricing Feed를 얹는 3-Layer 구조가 필수다.

🔄 비동기 수집 협업 모델로 재설계

설계사 SME가 상시 외근 상태라는 사실은 우리가 일하는 방식 자체를 바꿔야 함을 의미한다. '설계회사에 묻는다'가 아니라 '설계회사가 평소 만들어내는 결과물을 수동으로 수집·라벨링·학습'하는 비동기 모델로의 전환이다.

WISE42는 '철도전기 도메인의 5년치 암묵지를
디지털 자산으로 전환하고 지속적으로 갱신하는 운영체계'를 제공한다.
이것이 과거 실패한 SW 사업자들과 우리를 가르는 본질적 차이다.
솔루션 아키텍처

3-Layer Architecture —
LLM 단독의 한계를 넘어서

본 프로젝트의 본질적 난이도를 고려할 때, 단일 LLM 호출만으로는 절대 해결되지 않는다. 다음과 같은 3-Layer 아키텍처가 필수다.

Layer 1 · 선행 필수
🗂️ Ontology Layer — 자재·공종 마스터
역할: 철도전기 도메인 분류체계 구축
  • 철도전기 자재 1만+ SKU에 대한 분류체계 구축
  • 변종 차원: 지역 / 궤도종류 / 운행속도 / 운행간격 / 전압 / 시공방식
  • 초기 시드데이터: 엔텍·세종기술의 과거 견적·산출서 기반
  • 핵심 원칙: Ontology는 모델보다 먼저 — 시드 분류체계가 부정확하면 어떤 모델도 정확하게 동작하지 않음
Layer 2
🔍 Extraction Layer — 도면 → 수량
역할: 도면 해독 및 수량 산출
  • Multi-modal LLM(도면 이해) + Vision OCR + 도면 기호/심볼 인식 + 도면-자재 매핑
  • 준공도면 우선 학습(클린 데이터셋) — '준공도면 우선 학습' 원칙
  • 설계 중 도면은 수정 이력과 함께 학습하여 변경 패턴까지 예측
  • 불확실성의 가시화 — 모델이 자신없는 항목을 사람 검수 항목으로 자동 분류하여 SME 부담 정량화
Layer 3
💰 Pricing Layer — 단가 산정
역할: 공사비 산출을 위한 단가 관리
  • 자재별 견적 이력 DB + 공급사 견적 자동 요청 워크플로우
  • 6개월 가격변동 대응: 단가 신선도(freshness) 스코어를 시스템이 자체 표시
  • 견적 미보유 자재 발생 시 자동 RFQ 발송 → 회신 결과 학습 — 단가 미보유는 '오류'가 아니라 'RFQ 자동발송 이벤트'로 정의

Living System 운영 모델

과거 실패 패턴과의 결별 —
4가지 운영 원칙

💳 구독형(SaaS) 단가 체계

1회 라이선스 판매 명시적 금지. 연간 운영료 + 사용량 기반 구조로만 제안. 초기 도입 비용을 낮추고, 가치가 검증된 후 단계적 확장.

🔄 자동 업데이트 파이프라인

신규 견적·신규 도면이 들어올 때마다 모델·DB 자동 재학습. 단가 신선도 스코어 일별 갱신, Ontology 변경 분기별 정기 릴리스.

🏛️ 공단 내부 운영팀 양성

KRNA 전철전력처 내 2~3명을 시스템 운영 담당자로 지정·교육. 외부 의존을 점진적으로 줄여, 본 시스템이 공단 내부 자산으로 정착하도록 설계.
