# Software 3.0 & 에이전틱 엔지니어링

## 한 줄 정의
> LLM을 인터프리터로, 컨텍스트 윈도우를 프로그램으로 삼는 새로운 소프트웨어 패러다임. 코드를 '쓰는' 것에서 에이전트를 '지휘하는' 것으로의 전환.

## 소프트웨어 3세대 진화

| 세대 | 프로그래밍 방식 | 핵심 레버 | 시대 |
|------|-------------|---------|------|
| **Software 1.0** | 인간이 명시적 코드를 작성 | 코드 파일 | ~2010년대 |
| **Software 2.0** | 데이터셋·목적함수·신경망 설계 | 학습된 가중치(Weights) | 2010년대~ |
| **Software 3.0** | 프롬프트·컨텍스트·도구·메모리로 LLM을 프로그래밍 | **컨텍스트 윈도우** | 2025년~ |

> Software 3.0에서는 LLM이 인터프리터이고, 컨텍스트 윈도우가 프로그램이다.

## Vibe Coding vs 에이전틱 엔지니어링

| 구분 | Vibe Coding | Agentic Engineering |
|------|------------|---------------------|
| **목표** | 바닥을 높인다 (Floor ↑) | 천장을 높인다 (Ceiling ↑) |
| **대상** | 누구나 소프트웨어를 만들 수 있게 | 전문가가 품질을 유지하며 10배 빠르게 |
| **결과물** | 프로토타입·개인 도구 | 프로덕션 수준 시스템 |
| **인간의 역할** | 방향 제시 | 스펙 설계 + 감독 + 품질 검증 |

## 에이전틱 엔지니어링의 핵심 스킬

```
1. 에이전트를 위한 명확한 스펙(Spec) 작성
2. 생성된 코드 diff 검토 및 품질 판단
3. 테스트·평가 루프(Eval Loop) 설계
4. 보안 경계 및 권한 관리
5. 시스템 전체 구조에 대한 이해
```

## 검증가능성(Verifiability) 프레임워크

> 전통 소프트웨어는 **명세(specify)할 수 있는 것**을 자동화한다.
> LLM은 **검증(verify)할 수 있는 것**을 자동화한다.

```
능력 향상 속도 ≈ 검증가능성 × 학습 집중도 × 데이터 커버리지 × 경제적 가치
```

**AI가 빠르게 잘하게 된 영역**: 수학, 코딩, 테스트, 게임 (→ 검증이 자동화됨)
**아직 느린 영역**: 미학, 시스템 설계, 창의적 판단 (→ 보상 신호 설계가 어려움)

## Jagged Intelligence (들쭉날쭉한 지능)

현재 AI의 본질적 특성: 특정 능력은 인간을 초월하고, 인접 능력은 초등학생만도 못하다.

**예시**: 10만 줄 코드베이스 리팩토링 가능 ↔ "50m 떨어진 세차장, 걸어가야 하나 차로 가야 하나?" 질문에 오답

**원인**: RL(강화학습) 보상이 집중된 영역 vs 데이터 분포에서 소외된 영역의 격차

**실전 함의**: 모델이 어떤 "회로"에서 잘 동작하는지 파악하는 것이 AI 활용의 핵심

## "앱이 사라진다" — MenuGen 사례

기존 접근: 사진 → OCR → 이미지 생성 API → UI 렌더링 → 배포 (복잡한 앱 스택)
Software 3.0 접근: 사진 + 프롬프트 → 멀티모달 모델이 직접 이미지 위에 결과 렌더링

> 이 앱은 존재해선 안 된다. 신경망이 직접 처리한다.

**창업자·기획자에게**: 기존 앱을 "더 빠르게" 만드는 것만 생각하지 마라. 아예 앱이 없어야 하는 경우를 먼저 찾아라.

## LLM Knowledge Base (LLMwiki의 이론적 근거!)

Karpathy가 직접 설명한 새로운 정보 처리 카테고리:

```
기존 RAG: 원문 → 검색 → 답변 (매번 원문 조회)
LLM Wiki: 원문들 → AI가 지속적 컴파일 → 구조화된 위키
          (요약, 엔티티 페이지, 개념 페이지, 교차 링크, 모순 정리)
```

> "고전적 프로그램은 이런 지식 베이스를 만들 수 없었다. 하지만 LLM은 가능하다."

이것이 **지금 이 LLMwiki를 만드는 이유**와 정확히 일치한다.

## 인간이 더 가치 있어지는 역량

에이전트가 많은 것을 대체할수록 이것들이 더 희소해진다:
- **이해(Understanding)** — "생각은 위임할 수 있어도 이해는 위임할 수 없다"
- **취향(Taste)** — 코드·시스템의 미학적 품질 판단
- **보안 경계 설계** — 시스템이 침해당하지 않도록 구조 설계
- **"무엇을 만들 가치가 있나"** — 방향 설정과 우선순위 판단

## 채용 방식의 변화

기존: 코딩 퍼즐 → **에이전틱 엔지니어링 실력 테스트로 전환 필요**

새로운 면접 형식 (Karpathy 제안):
```
"대규모 프로젝트를 에이전트로 구현하고 배포하라.
그 후 10개의 공격 에이전트가 당신의 시스템을 뚫으려 한다.
막을 수 있는가?"
```
