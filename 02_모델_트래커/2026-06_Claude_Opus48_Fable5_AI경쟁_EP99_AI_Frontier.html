3대 핵심 개선사항
🎯
정직성 (Honesty) 강화
Opus 4.8의 가장 주목할 변화. 코드 결함을 사용자에게 보고하지 않는 '과잉 순응(sycophancy)' 행동이 Opus 4.7 대비 4배 감소. 에이전트가 장기 실행될수록 인간이 체크하는 횟수가 줄어드는데, 이때 AI의 정직성이 결과 품질을 좌우한다.
"발견된 버그를 말하지 않는 AI는 디버깅보다 더 큰 비용을 만든다"
🔀
Dynamic Workflows
수십만 줄 규모의 코드베이스 마이그레이션 등 거대 작업을 AI가 자동으로 병렬 분할. 수백 개의 서브에이전트를 동시 실행하는 구조를 관리. 인간이 설계할 수 없는 복잡도의 워크플로우를 AI가 스스로 아키텍처링.
Code as harness: 코드가 결과물이 아니라 에이전트를 묶는 뼈대가 된다
⚡
Effort Control
claude.ai와 Cowork 환경에서 응답 '노력 수준'을 조절 가능. 기본값은 high. 빠른 답변이 필요한 경우 low/medium으로 낮춰 latency·비용 최적화. Adaptive Thinking(적응형 사고)은 상시 활성화.
운용 비용과 응답 품질의 트레이드오프를 조직이 직접 제어
🤝
Managed Agents
기업 경계 내에서 에이전트를 안전하게 실행하는 관리 계층. 외부 클라우드에 데이터가 나가지 않고 기업 인프라 내에서 자율 에이전트 워크플로우를 구현. Enterprise Agentic Platform 전쟁에서 Anthropic의 핵심 무기.
데이터 주권 + 에이전트 자율성의 동시 달성 시도

EP99 핵심 논점: Google I/O에서 Gemini 3.5 Flash가 "프론티어 성능 + 저지연"을 표방하며 출시. 토큰 처리량이 6.6배 폭증한 것은 기업 도입 규모가 이미 새로운 차원에 진입했음을 의미. 이제 경쟁은 모델 점수 0.x% 개선이 아니라, 얼마나 싸게·빠르게·안전하게 대규모 실행하느냐의 싸움.
EP99 주요 논점 정리
🚀
AI 모델 발전 주기 가속
Opus 4.8, Fable 5, Gemini 3.5, GPT-5.5가 수주 간격으로 출시되는 시대. 2025년까지는 6개월 사이클이었다면 2026년은 수주~1개월 단위. 기업 의사결정 속도가 AI 출시 주기를 따라가지 못하는 구조적 문제 발생. "우리가 도입을 검토하는 사이 모델이 두 번 바뀐다."
🔧
Code as Harness — 코드는 에이전트의 뼈대다
EP99에서 노정석이 강조한 개념. 소프트웨어 3.0 시대에서 코드는 더 이상 최종 결과물이 아니라 에이전트들을 연결하는 하네스(harness, 뼈대·장치). 수십~수백 개의 서브에이전트가 병렬 실행될 때, 코드는 그 흐름을 제어하는 오케스트레이션 레이어가 된다. Dynamic Workflows는 이 패러다임의 첫 번째 상용 구현.
⏱️
장기 실행 에이전트: 운용 구조가 진짜 경쟁력
단발성 질문-응답을 넘어, 수시간~수일에 걸쳐 자율 실행되는 에이전트가 실제 산업에 투입 중. 이때 모델 점수보다 중요한 것은 ①비용 예측 가능성, ②latency 안정성, ③실패 복구 메커니즘, ④인간 개입 지점 설계. Enterprise Agentic Platform 전쟁의 본질.
👤
인간의 일이 바뀐다 — 주의력 자원의 재배분
에이전트가 루틴 업무를 대체할수록, 인간의 희귀 자원은 '시간'이 아닌 '주의력(attention)'. Brockman(OpenAI)이 언급한 "인간 주의력이 병목"과 같은 맥락. 조직 내 AI화가 깊어질수록, C레벨의 집중 영역과 의사결정 빈도가 근본적으로 달라져야 한다. "AI를 더 잘 쓰는 인간"이 아니라 "AI가 실행하는 동안 인간이 무엇을 할지" 설계가 핵심.
GARAMbada 인사이트 (WISE42 관점)
인사이트 01
Opus 4.8 = 즉시 활용 가능한 수준
SWE-bench Pro 69.2% + 코드 결함 미보고 4배 감소는 WISE42의 KRNA 프로젝트처럼 '검증 정확도'가 핵심인 B2B 에이전트에 직접적 영향. 정직성 강화는 단순 성능 향상이 아니라 신뢰 기반 자동화의 기반.
Managed Agents로 KRNA 에이전트 아키텍처 업그레이드 검토
인사이트 02
Fable 5 = 새 가격 구간 진입 검토
Fable 5가 $10/$50으로 GA 공개됨에 따라, Opus 계열($5/$25)과 Fable 계열($10/$50) 사이 실질 성능 차이 검증이 필요. 운용 비용이 2배인 만큼 ROI 기준을 명확히 설정해야 함. 초대 전용 Mythos 5는 당분간 관망.
Fable 5 vs Opus 4.8 A/B 테스트 설계
인사이트 03
Dynamic Workflows = Founders Studio 핵심 인프라
수백 개 서브에이전트 병렬 실행 구조는 WISE42 Founders Studio의 '다사업 동시 운영' 가설과 직결. 사람이 조율하던 업무 흐름을 Dynamic Workflows로 대체하면 창업자 1인이 관리 가능한 사업 수가 현실적으로 확대된다.
Founders Studio MVP에 Dynamic Workflows 아키텍처 적용 검토
인사이트 04
Code as Harness → FDE 설계 원칙 업데이트
코드가 '결과'가 아닌 '에이전트 뼈대'라는 관점은 FDE(Field Deployment Engine) 설계 원칙에 통합이 필요. 기능 구현 코드를 짜는 것이 아니라 에이전트 실행 흐름을 정의하는 오케스트레이션 레이어를 설계하는 역량이 핵심.
FDE 기술 스택에 에이전트 오케스트레이션 레이어 추가
인사이트 05
모델 선택 = 비용·latency 최적화 문제
Haiku($1/$5) → Sonnet($3/$15) → Opus($5/$25) → Fable($10/$50) 4단계 계층에서 각 업무 유형별 최적 모델 매핑이 실질 운영 비용을 3~10배 차이 낸다. Effort Control을 활용하면 동일 모델 내에서도 비용·품질 트레이드오프 가능.
업무 유형 × 모델 매핑표 작성 및 월별 비용 모니터링 체계 구축
인사이트 06
주의력 감사 (Attention Audit) 우선
EP99의 "인간의 일 변화" 논점은 WISE42 내부에도 적용 가능. 에이전트가 처리하는 업무가 늘어날수록, C레벨이 집중해야 할 '주의력 영역'을 명시적으로 정의하지 않으면 관리 비용은 줄지 않고 의사결정 품질만 낮아지는 역효과 발생.
