# 나만의 LLMwiki 만들기 — 단계별 가이드

> 대상: LLM을 비즈니스에 활용하는 C-level / 의사결정자
> 목표: 흩어진 LLM 지식을 체계적으로 정리하고, 빠르게 꺼내 쓸 수 있는 나만의 지식 베이스 구축

---

## 왜 LLMwiki가 필요한가?

LLM 생태계는 3~6개월 주기로 판도가 바뀐다. 논문, 모델 출시, 규제, 스타트업 M&A가 동시다발적으로 쏟아지는 상황에서 C-level이 직접 모든 것을 따라가기는 불가능하다. LLMwiki는 다음 세 가지 문제를 해결한다.

- **의사결정 속도**: "이 모델이 우리 유스케이스에 맞나?" 를 5분 안에 답할 수 있어야 한다.
- **팀 얼라인먼트**: 내가 정리한 기준이 팀 전체의 공통 언어가 된다.
- **누적 자산화**: 매번 새로 공부하지 않고, 지식이 쌓이면서 복리로 작동한다.

## STEP 2 — 콘텐츠 전략: 무엇을 담을 것인가?

### 2-1. 중급자가 반드시 정리해야 할 핵심 개념

#### 🔹 모델 이해

- **파라미터 수 vs 성능**: 파라미터가 많다고 무조건 좋은 게 아닌 이유
- **컨텍스트 윈도우**: 128K, 1M 토큰의 실제 의미와 비용 임팩트
- **멀티모달**: 텍스트·이미지·음성·코드를 함께 처리하는 모델의 비즈니스 가치

#### 🔹 아키텍처 패턴

| 패턴 | 언제 쓰나 | 핵심 트레이드오프 |
|------|-----------|-----------------|
| **RAG** (검색증강생성) | 사내 문서 기반 QA | 최신성 ↑, 환각 ↓ / 인프라 복잡도 ↑ |
| **Fine-tuning** | 특정 도메인 어조·형식 고정 | 성능 ↑ / 비용·시간 ↑ |
| **Function Calling** | 외부 시스템 연동 자동화 | 유연성 ↑ / 오류 처리 난이도 ↑ |
| **Multi-Agent** | 복잡한 워크플로우 분업 | 확장성 ↑ / 디버깅 난이도 ↑↑ |

#### 🔹 비용 구조

```
비용 = (입력 토큰 × 입력단가) + (출력 토큰 × 출력단가)

실전 팁:
- 출력 토큰이 입력 토큰보다 2~5배 비싸다
- 캐싱(Prompt Caching)으로 반복 비용 90% 절감 가능
- 배치 처리(Batch API)로 비실시간 작업 50% 절감
```

### 2-2. 문서 한 장의 표준 템플릿

```markdown
# [주제명]

## 한 줄 정의
(30자 이내로 핵심만)

## 왜 중요한가?
(비즈니스 임팩트 중심으로 2~3문장)

## 핵심 내용
(개념, 작동 원리, 주요 지표)

## 실전 활용
(우리 회사/업무에 적용 방법)

## 주의사항 / 함정
(흔한 오해, 실패 패턴)

## 참고 자료
(링크, 날짜 포함)

## 업데이트 이력
- YYYY-MM-DD: 최초 작성
```

## STEP 3 — 도구 선택: 어디에 쌓을 것인가?

### 3-1. 도구 비교

| 도구 | 강점 | 약점 | 추천 대상 |
|------|------|------|-----------|
| **Obsidian** | 로컬 저장, 링크 그래프, 마크다운 | 팀 공유 불편 | 개인 깊이 있는 정리 |
| **Notion** | 팀 공유, 데이터베이스 뷰 | 검색 성능, 오프라인 | 팀과 함께 쓸 때 |
| **GitHub + Markdown** | 버전 관리, 개발팀 친화 | 비개발자 진입장벽 | 개발팀과 협업 |
| **Claude Cowork** | AI가 직접 문서 작성·업데이트 지원 | 현재 베타 단계 | AI 협업형 위키 **(현재 선택)** |

> 현재 이 프로젝트는 **Claude Cowork**를 활용한 AI 협업형 LLMwiki로 구성되어 있습니다.
> Claude가 리서치→작성→업데이트를 함께 수행합니다.

### 3-2. Claude Cowork 활용 워크플로우

```
[인풋] 읽은 논문 / 뉴스 / 회의 메모
    ↓
[Claude에게 요청] "이 내용을 LLMwiki 형식으로 정리해줘"
    ↓
[Claude 작업] 관련 파일 업데이트 or 새 문서 생성
    ↓
[내가 검토] 인사이트 한 줄 추가
    ↓
[누적] 위키가 점점 나를 닮아감
```

## STEP 5 — 운영 & 유지보수: 위키를 살아있게 만들기

### 5-1. 문서 신선도 관리

| 카테고리 | 업데이트 주기 | 이유 |
|----------|-------------|------|
| 기초개념 | 분기 1회 | 변화 느림 |
| 모델 비교 | 월 1회 | 새 모델 지속 출시 |
| 비즈니스 활용 | 월 2회 | 실전 경험 누적 |
| 트렌드 레이더 | 주 1회 | 빠른 변화 |
| 나의 인사이트 | 수시로 | 즉흥적 아이디어 포착 |

### 5-2. 위키가 죽는 3가지 이유와 해결책

| 실패 패턴 | 원인 | 해결책 |
|-----------|------|--------|
| "나중에 정리해야지" | 정리 시점 미룸 | 30분 내 초안 먼저, 완성도는 나중 |
| 너무 완벽하게 쓰려 함 | 완벽주의 | 70% 완성도로 일단 발행 |
| 혼자만 쓰는 문서 | 활용 안 됨 | 팀 미팅에서 위키 문서 직접 인용 |

### 5-3. C-level을 위한 "1분 리뷰" 습관

매일 5분, 이 세 가지만 체크:
1. **오늘 들은 LLM 관련 키워드** → 용어사전에 추가
2. **오늘 내린 AI 관련 결정** → 의사결정 로그에 한 줄
3. **오늘 발견한 흥미로운 정보** → 해당 카테고리 파일에 링크
