01 ·

패러다임 시프트 — 생성에서 추론으로

2022~2024년이 '생성(Generation)'의 시대였다면, 2025년 이후의 공공·산업 AI는 '추론(Reasoning)'과 '실행(Action)'의 시대로 이동하고 있다. 그 분기점이 온톨로지 기반 AI다.

Mode A · 기존
Generative LLM

확률적 토큰 예측을 통해 "다음에 올 가장 그럴듯한 말"을 만든다. 학습된 패턴의 통계적 재구성이다.

유창함이 강점이지만, 사실성·일관성·감사가능성은 부산물에 불과하다.

→ P(token | context) 최적화

VS
Mode B · 제안
Ontology-Grounded AI

현실 세계의 객체·관계·규칙을 명시적으로 모델링한 지식 그래프 위에서 추론한다.

판단 근거를 트레이싱 가능하고, 규정 위반 시 거부할 수 있다. 책임을 질 수 있는 AI다.

→ Reason(Entity, Relation, Rule) 실행

Insight · WISE42 Thesis 공공·국방·인프라 영역에서 LLM 단독 배포는 본질적으로 부적합하다. 책임 소재(accountability)와 감사 추적(audit trail)이 불가능하기 때문이다. 온톨로지는 LLM을 '현실의 규칙'에 묶어두는(grounding) 닻이며, 이것이 행정 운영체제(Governance OS)의 본질이다.
02 ·

아키텍처 해부 — 지식 그래프의 구조

온톨로지 기반 AI는 단순한 데이터베이스나 검색 시스템이 아니다. 그것은 행정 현실을 컴퓨터가 추론할 수 있는 형태로 번역해놓은 '의미망(Semantic Network)'이다.

온톨로지의 4계층 구조

Layer 1 · Entities
객체 (Things)
시민, 차량, 건물, 정책, 부서, 사건, 시설 등 행정이 다루는 실세계의 모든 명사. 각 객체는 고유 ID와 속성 집합을 갖는다.
Layer 2 · Relations
관계 (Edges)
"거주한다", "관할한다", "위배한다", "발생했다" 등 객체 간 의미적 연결. 방향성과 시간 속성을 갖는 1차 술어 관계다.
Layer 3 · Rules
규칙 (Axioms)
"폭우 + 침수이력 + 노인가구 → 즉각대응 대상" 같은 추론 규칙. SWRL/SHACL로 인코딩되며, 행정 규정이 이 계층에 산다.
Layer 4 · Actions
행동 (Operators)
'알림 발송', '예산 집행 요청', '담당자 배정' 등 실제 행정 시스템에 발화 가능한 실행 가능 행위. 워크플로우 엔진과 연결된다.

실제 지식 그래프 예시 — 재난 대응 시나리오

FIG · 2.2
Disaster Response Knowledge Graph

위 그래프에서 보듯, 부서별로 흩어진 데이터(기상청의 강수량, 도시계획과의 침수이력, 복지과의 가구 통계)가 단일 의미망으로 연결된다. LLM에게는 별개의 텍스트일 뿐인 정보가, 온톨로지에서는 하나의 DECIDE 노드로 수렴한다.

ontology / disaster.ttl Turtle / RDF
# 규칙 (Layer 3): 추론 엔진이 자동 실행 Rule UrgentResponseTrigger: IF Event(?e, type=Flood) AND Location(?l, hasHistory=?h) AND ?h.count >= 2 AND District(?l).elderlyRatio > 0.20 THEN Trigger(action="IMMEDIATE_DISPATCH", target=?l, priority="CRITICAL", audit="§시행령14-3")

이 규칙은 LLM의 'attention weight'처럼 블랙박스가 아니다. 읽을 수 있고, 디버깅할 수 있으며, 법령 조항과 1:1로 매핑된다. 이것이 행정에서 요구되는 '감사 가능한 자동화'의 본질이다.

03 ·

추론 엔진 — Symbolic × Neural의 결합

오해를 풀자. 온톨로지 기반 AI는 LLM을 '대체'하지 않는다. LLM을 '경계 짓는다'. 양자는 다른 임무를 수행하는 보완재다.

01
사용자 자연어 질의 입력
"강남역 일대 폭우 대응 계획 보여줘" — 비정형 한국어 자연어.
Input · Natural Language
02
LLM이 의도를 온톨로지 쿼리로 번역
LLM은 답을 '생성'하지 않는다. 자연어를 SPARQL/Cypher 쿼리로 변환하는 '번역기' 역할만 수행한다.
LLM as Translator · Not Generator
03
지식 그래프에서 사실(Facts) 조회
실제 행정 DB·센서 데이터에 연결된 그래프에서 해당 객체와 관계를 추출한다. 여기서 '환각'은 구조적으로 불가능하다.
Knowledge Graph · Ground Truth
04
규칙 엔진이 추론 및 의사결정 수행
SWRL/SHACL 기반 추론기가 행정 규정에 따라 결정을 내린다. 각 결정에는 적용된 규칙 ID가 부착된다.
Symbolic Reasoner · Auditable
05
LLM이 결과를 자연어로 재가공
추론 결과(JSON/RDF)를 사람이 읽기 좋은 보고서·시각화로 변환. 이 단계에서도 LLM은 사실을 '추가'하지 않고 '표현'만 한다.
LLM as Presenter · Bound
06
실행 행위 발화 또는 사람 결재 요청
결정의 위험도(risk score)에 따라 자동 실행하거나 인간 결재자에게 라우팅. Human-in-the-loop이 설계의 핵심.
Action / Approval Gate
Common Misconception "온톨로지 AI는 LLM이 없어서 안 똑똑하다"는 오해가 흔하다. 사실은 정반대다. LLM의 유창함을 쓰되, 사실 판단은 결정론적 그래프 엔진에 위임한다. 이것이 'Neuro-Symbolic AI'의 핵심이며, Palantir AIP·DARPA XAI 프로그램이 채택한 표준 아키텍처다.
04 ·

신뢰 모델 — Hallucination부터 xAI까지

공공 영역에서 AI 도입의 가장 큰 장벽은 '왜 그렇게 판단했느냐'는 질문에 답할 수 없다는 점이다. 온톨로지는 이 질문에 정면으로 답한다.

차원 Generative LLM Ontology-Based AI
Hallucination Rate 3~27% (GPT-4 기준, 도메인 의존) 구조적 ≈ 0% — 그래프 외 사실 생성 불가
Explainability Black Box — Attention Map으로도 인간 해석 불가 White Box — 적용된 규칙 ID·근거 노드 추적 가능
Audit Trail 없음. 로그는 입출력 텍스트만 보존. 완전 — 모든 추론 단계 RDF 트리플로 저장
Regulation Compliance 법령 변경 시 재학습 필요. 정합성 보장 불가. 규칙 1줄 수정으로 즉시 반영. 정합성 검증 가능.
Adversarial Robustness 프롬프트 인젝션·탈옥 공격에 취약 스키마 위반 입력은 거부. 공격면 축소.
Update Cycle 파인튜닝 = 수일~수주, GPU 비용 발생 규칙·온톨로지 편집 = 분 단위, CPU 추론

설명 가능성(xAI)의 실제 작동

audit_trace.json — 단일 의사결정의 감사 흔적 Output
{ "decision_id": "DEC-2026-05-22-0014", "action": "IMMEDIATE_DISPATCH", "target": "강남역_침수위험구역", "confidence": 0.94, "rules_applied": [ "R-FLOOD-001 (시행령 §14조3항)", "R-ELDERLY-007 (재난기본법 §22조)" ], "evidence_facts": [ "기상청.강수량(2026-05-22T15:00) = 112mm/h", "도시계획.침수이력(강남역) = [2022, 2024]", "통계청.노인가구비율(강남구) = 0.27" ], "human_reviewer": "방재과장_OOO", "reviewed_at": "2026-05-22T15:03:42Z" }

이 JSON은 행정 감사관, 의회, 시민 누구든 열어볼 수 있다. "왜 그렇게 결정했는가"에 대한 답이 코드가 아닌 법령 조항으로 제시된다. 이것이 공공 AI가 도달해야 할 신뢰의 표준이다.

05 ·

소버린 AI — 데이터 주권의 기술적 실현

공공·국방 데이터를 OpenAI 서버로 보내는 것은 단순한 보안 문제가 아니라 주권의 문제다. 온톨로지 기반 AI는 설계 단계부터 폐쇄망 운영을 전제한다.

Topology
완전 폐쇄망 배포
관용 클라우드(K-Cloud) 또는 기관 자체 데이터센터 내에 On-Premise로 설치. 외부 인터넷 연결 없이 동작 가능.
Model
국내 LLM 결합
Upstage Solar, LG EXAONE 등 국내 sLLM을 온톨로지 위에서 운영. 한국어·한국 법령 도메인에 최적화.
Governance
데이터 출국 0
학습 데이터·추론 로그·사용자 입력 어떤 것도 국외로 전송되지 않음. GDPR·개인정보보호법 자동 준수.
Resilience
독립 운영
외부 API 의존성 제로. OpenAI 서비스 장애·정책 변경·지정학 리스크에 영향 없음.
Reference · WeBIG.ai 사례 WISE42의 WeBIG.ai는 KRNA, FIPA 등 공공기관 배포에서 폐쇄망 온프레미스 설치를 표준으로 한다. 기관 IDC 내부에서 LLM 추론·온톨로지 추론·실행 액션이 모두 닫힌 경계 안에서 발생하며, 이는 'Sovereign AX'의 실증 모델이다.
06 ·

하이브리드 스택 — 실전 구현 청사진

현실의 행정 AI는 '온톨로지냐 LLM이냐'의 선택이 아니다. 양자를 어떻게 배치하느냐의 설계 문제다. 다음은 WISE42 표준 7계층 스택이다.

FIG · 6.1
WISE42 AX Operating Stack (7-Layer)

핵심은 L4·L5(Ontology + Reasoning)가 시스템의 무게중심이라는 점이다. LLM(L6)은 입출력 인터페이스일 뿐이며, 교체 가능하다(Solar→EXAONE→Llama). 반면 온톨로지와 규칙은 기관의 도메인 자산이자 영속적 IP가 된다.

07 ·

의사결정 매트릭스 — 언제 무엇을 쓸 것인가

모든 업무에 온톨로지가 필요한 것은 아니다. 다음 매트릭스는 C-level 도입 결정에 직접 사용할 수 있다.

Quadrant I · 저위험 · 비정형
민원 1차 응대
회의록 요약
→ 생성형 LLM 단독으로 충분. 인간 검토 후 외부 발신.
Quadrant II · 고위험 · 규정기반
재난 대응 의사결정
복지 사각지대 판정
국방·SCADA 운영
→ 온톨로지 기반 AI 필수. LLM은 보조적 NLU/NLG로만.
Quadrant III · 저위험 · 정형
통계 보고서 생성
FAQ 자동화
→ LLM + RAG로 처리. 온톨로지는 ROI 낮음.
Quadrant IV · 고위험 · 비정형
정책 영향 시뮬레이션
예산 우선순위 추론
→ 하이브리드(온톨로지 + LLM). WISE42 Harness 표준 영역.
Closing · Operating System Thesis 서울시 AX 제안서가 말하는 온톨로지 기반 AI는 단순한 챗봇이 아니다. 그것은 복잡한 행정 규정과 데이터를 이해하고, 실제 업무를 수행하며, 그 결과를 설명할 수 있는 '행정 운영체제(Governance OS)'다. 윈도우와 리눅스가 PC 시대를 정의했듯, 온톨로지 기반 AI는 공공 AX 시대의 OS다. 그리고 그 OS의 한국형 표준을 만드는 것이 WISE42의 사명이다.
