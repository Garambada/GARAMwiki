01 — Core Architecture

Prefill vs Decode — 두 개의 다른 물리 세계

LLM 추론은 하나의 프로세스가 아니다. 완전히 다른 두 단계가 연속된 것이며, 각각의 병목은 완전히 다른 곳에 있다.

Prefill

입력 프롬프트 전체를 동시에 처리하고 KV Cache를 채우는 단계

  • 입력 토큰 전부를 병렬로 처리 가능
  • GPU의 행렬 연산(FLOP/s)이 한계 결정
  • 배치 크기를 키울수록 효율 향상
  • KV Cache 완성 → Decode 시작 신호
Decode

토큰 하나씩 순차적으로 생성하는 단계 — 출력 토큰이 비싼 이유

  • 토큰 1개 생성마다 KV Cache 전체를 읽어야 함
  • HBM 대역폭(3.35 TB/s)이 한계 결정
  • 컨텍스트 길수록 읽는 데이터 선형 증가
  • GPU FLOPs의 90%+ 유휴 상태
Memory Time (Decode 지배)
t_mem = 메모리 접근 바이트
÷ HBM 대역폭 (TB/s)
70B 모델 BF16 = ~140GB → H100 기준 약 42ms/token
Compute Time (Prefill 지배)
t_compute = 필요 FLOPs
÷ GPU FLOP/s
입력이 길수록 FLOPs 비례 증가, 배치 클수록 효율
실제 처리 시간 = max( t_mem , t_compute )  —  Roofline 분석의 핵심
GPU HBM 대역폭 FP16/BF16 FLOP/s Decode 상한 (70B) 세대
H100 SXM 3.35 TB/s 989 TFLOP/s ~24 tok/s (단일 GPU) Hopper
B200 (Blackwell) ~8 TB/s ~4,500 TFLOP/s ~57 tok/s (단일 GPU) Blackwell
02 — KV Cache

KV Cache — HBM을 먹는 메모리 괴물

Transformer의 모든 주의(Attention) 계산 결과를 저장하는 KV Cache는 Decode 성능과 비용의 핵심 변수다.

📝
입력 프롬프트
시스템 프롬프트 + 사용자 메시지 + 컨텍스트 문서
⚡
Prefill 처리
입력 전체 병렬 처리
각 레이어·헤드 K, V 계산
🗄️
KV Cache 저장
HBM에 Key-Value 행렬 저장
70B+128K = 수십 GB
🔄
Decode 루프
토큰 1개 생성마다
KV Cache 전체 읽기 반복
✅
출력 완성
EOS 토큰 또는
max_tokens 도달 시 종료
⚠️ 왜 Output 토큰이 3~10배 비싼가?

Decode 단계에서 토큰 1개를 만들 때 GPU는 KV Cache 전체(수십 GB)를 읽어야 한다. 실제 FLOPs는 토큰 1개 분밖에 없지만 메모리 I/O는 KV Cache 전체 크기만큼 발생 → GPU의 FLOP 활용률 극히 낮음 → 같은 GPU로 처리할 수 있는 출력 토큰 수가 입력 토큰 수보다 훨씬 적음 → 가격 높게 책정.

PagedAttention (vLLM의 핵심)

OS의 가상 메모리 페이지 방식을 KV Cache에 적용. 기존 방식은 각 요청마다 연속된 메모리 블록 예약 → 단편화로 40~60% 낭비. PagedAttention은 비연속 페이지에 KV Cache 저장 → 메모리 활용률 극대화 → 동시 처리 요청 수(throughput) 2~4배 향상.

03 — Token Economics

토큰 가격 구조 — 물리학이 만든 가격

API 가격표의 숫자들은 임의적이지 않다. GPU의 물리적 제약이 가격 계층을 결정한다.

긴 컨텍스트는 KV Cache가 GPU 메모리를 초과할 수 있음. 일부 제공업체가 200K 이상에서 추가 과금. 장문서 처리 앱 설계 시 핵심 비용 변수.

💡 비용 최적화 우선순위

① Output 토큰 수 최소화 (가장 비쌈) → ② Prompt Caching 도입 → ③ 배치 API 활용 → ④ 입력 청크 최적화 순서로 접근.

04 — Queue Theory

Erlang-C — GPU 플릿을 몇 개나 사야 하는가

GPU 플릿 규모는 평균 수요가 아닌 수요의 변동성(CV)이 결정한다. 이것이 자체 호스팅 경제학의 핵심이다.

변동계수(CV)와 필요 GPU 버퍼
CV = 0
평균량만 필요
CV = 0.3
버퍼 30~50%
CV = 0.5
버퍼 80~120%
CV = 1.0
2~3배 과잉
CV > 1
비용 폭발
자체 호스팅 손익분기점
CV < 0.3 ~ 0.4
대부분 기업의 실제 CV: 0.5 ~ 2.0 → API 사용이 경제적
실전 해석

Erlang-C는 대기열 이론의 공식으로, 서버 수·도착률·서비스율의 관계를 모델링한다. LLM 인프라에 적용하면:

입력 파라미터
λ = 평균 요청 도착률 (req/sec)
μ = GPU 1대의 처리율
σ = 수요 표준편차
핵심 결론
→ SLA(P95 지연) 충족 GPU 수 = f(λ, μ, σ/λ)
흔한 실수

평균 수요 기준으로 GPU를 구매하면 피크 시간에 큐가 폭발한다. CV 없이 ROI 계산하면 반드시 틀린다.

05 — API vs Self-hosting

수요 풀링 — API 사업자의 구조적 해자

왜 Anthropic이나 OpenAI의 GPU 이용률은 90% 이상인데, 기업 자체 호스팅은 50% 달성도 어려운가.

N명 고객의 수요를 통합 처리 시:
통합 수요 표준편차 = 개별 표준편차 × 1 / √N
N = 1,000 → 변동 1/32 축소  |  N = 10,000 → 변동 1/100 축소
API 고객 수 (N) 변동 감소율 실효 CV (개별 CV=1.0 가정) GPU 이용률 비고
개별 기업 (N=1) - 0.5 ~ 2.0 30~60% 과잉 프로비저닝
소규모 팀 (N=10) 1/3.2× ~0.3 ~65% 손익분기 진입
API 사업자 (N=1,000) 1/32× ~0.03 ~85% 높은 이용률
대형 API (N=10,000+) 1/100× <0.01 90%+ 구조적 우위
✅ 자체 호스팅이 유효한 경우
데이터가 외부 나갈 수 없는 규제 환경 (의료·금융·국방)
수요가 극도로 예측 가능한 배치 파이프라인 (CV < 0.3)
GPU 수천 개 이상 규모로 자체 풀링 가능한 초대형 기업
범용 API로 성능 충족 불가능한 도메인 특화 파인튜닝 필수 시
✗ 자체 호스팅이 비경제적인 경우
수요가 시간대·요일별로 크게 변동 (CV > 0.4)
GPU 전담 운영 인력 비용 포함 시 API 대비 비쌈
최신 모델 접근이 중요한 경우 (업데이트 지연)
인프라 운영 역량이 핵심 역량이 아닌 스타트업

서빙 프레임워크 선택은 워크로드 패턴에 따라 결정되어야 한다.
