Karpathy가 마지막으로 코드 수정한 횟수
핵심 요약

"저는 제 인생에서 프로그래머로서 이렇게 뒤처진다는 느낌을 받아본 적이 없습니다. 12월부터 에이전트가 생성한 코드 덩어리들이 그냥 맞게 나왔습니다. 마지막으로 수정한 게 언제인지도 기억이 안 납니다."

— Andrej Karpathy, Sequoia AI Ascent 2026

① 2025년 12월이 변곡점이었다 — 에이전틱 코딩 도구가 "유용하지만 불안정"에서 "믿고 맡길 수 있는" 수준으로 단계 전환이 일어났다.
② Software 3.0 — 코드 작성(1.0) → 데이터셋 설계(2.0) → 프롬프트로 LLM 프로그래밍(3.0). 컨텍스트 윈도우가 새 프로그램이다.
③ Vibe Coding ≠ Agentic Engineering — 전자는 바닥을 높이고, 후자는 천장을 높인다. 전문가의 새 핵심 역량은 에이전트 지휘다.

소프트웨어 3세대 진화
1.0
Software 1.0
명시적 코드
핵심 레버: 코드 파일
인간이 모든 로직을 직접 코드로 작성. if-else, for-loop, 함수 하나하나 명시.
2.0
Software 2.0
학습된 가중치
핵심 레버: 데이터셋 + 목적함수
데이터와 목표를 설계하면 신경망이 프로그램을 학습. 이미지 인식, 추천 시스템.
3.0
Software 3.0 ← 지금
컨텍스트 윈도우
핵심 레버: 프롬프트 + 컨텍스트
LLM이 인터프리터. 컨텍스트 윈도우가 프로그램. 자연어 지시로 디지털 세계를 프로그래밍.
Vibe Coding vs Agentic Engineering
관점 🎵 Vibe Coding ⚙️ Agentic Engineering
목표 바닥을 높인다 Floor ↑ 천장을 높인다 Ceiling ↑↑
대상 누구나 소프트웨어 제작 가능 전문가가 품질 유지하며 10배+ 빠르게
결과물 프로토타입 · 개인 도구 프로덕션 시스템 · 보안된 서비스
인간의 역할 방향 제시, 결과 확인 스펙 설계, diff 검토, 보안 경계, 시스템 판단
위험 보안 구멍, 기술 부채 에이전트의 "그럴듯한 오류" 간과
10x 효과 제한적 훨씬 초과 가능
2025년 12월 — 변곡점의 실체

Karpathy가 "한 번도 프로그래머로서 이렇게 뒤처진다고 느낀 적 없다"고 말한 이유:

2025년 11월 이전
  • ✗ 자주 수정 개입 필요
  • ✗ 큰 덩어리 코드는 믿기 어려움
  • ✗ 내가 코드의 80% 직접 작성
  • ✗ 도구라기보다 "고급 자동완성"
2025년 12월 이후
  • ✓ 덩어리가 그냥 맞게 나옴
  • ✓ 마지막 수정이 언제인지 기억 안 남
  • ✓ 에이전트가 80% 이상 담당
  • ✓ 진정한 "위임"이 가능해짐

→ 작업 단위가 "코드 한 줄 타이핑"에서 "기능 구현", "서브시스템 리팩토링", "라이브러리 리서치", "테스트 작성+실행+수정" 등 매크로 액션으로 이동.

Verifiability 프레임워크 — AI가 빠른 이유

전통 소프트웨어는 명세(specify)할 수 있는 것을 자동화한다.
LLM은 검증(verify)할 수 있는 것을 자동화한다.

능력 향상 속도 ≈ 검증가능성 × 학습 집중도 × 데이터 커버리지 × 경제적 가치

// 이 4가지가 모두 높은 영역 = 코딩, 수학, 게임
// 보상 신호가 없는 영역 = 미학, 시스템 설계, 창의적 판단

Labs가 RL 환경을 집중 구축한 영역에서 모델이 비약적으로 발전한다. 반대로 경제적 가치가 낮거나 검증이 어려운 도메인은 여전히 정체. 창업 기회 = 가치 있고 검증 가능하지만 Labs가 아직 주목하지 않은 영역.

Jagged Intelligence — 들쭉날쭉한 지능

✅ 현재 AI가 잘하는 것

  • 10만 줄 코드베이스 리팩토링
  • 제로데이 취약점 발견
  • 수학 올림피아드(IMO) 수준 풀이
  • 수십만 줄 코드 한 번에 파악
  • 여러 언어 동시 번역·작성
  • PyTorch dim/axis/keepdim 디테일 기억

❌ 아직도 실수하는 것

  • "50m 거리 세차장, 운전해야 하나?" 오답
  • 미학·취향 판단 (코드가 "예쁜지")
  • 시스템 경계 설계 (이메일 vs 유저 ID 혼용)
  • 제품 방향성 판단
  • 보안 경계 직관적 이해
  • 코드 극도 단순화 (microGPT 스타일)

→ 모델이 어느 "RL 회로"에 있는지 파악하는 것이 AI 활용의 핵심. 내 작업이 그 회로 안이면 날고, 밖이면 기대 이하.

MenuGen — "이 앱은 존재해선 안 된다"

음식 메뉴 사진을 찍으면 각 음식의 이미지를 보여주는 앱. Software 3.0이 어떻게 앱 자체를 없애는지 보여주는 사례.

에이전트 시대 — 인간이 더 가치 있어지는 것들
🧠
이해 (Understanding)
"생각은 위임할 수 있어도, 이해는 위임할 수 없다." 방향 설정과 무엇을 만들 가치가 있는지 판단.
🎨
취향 (Taste)
코드·시스템의 미학적 품질 판단. RL 보상에 포함되지 않아 여전히 인간 영역. (에이전트 코드는 종종 "지저분하다")
🔒
보안 경계 설계
에이전트는 Stripe 이메일 ≠ Google 로그인 이메일을 혼동했다. 시스템 경계 직관은 여전히 인간이 담당.
🗺
방향 설정 (Direction)
"무엇을 만들 가치가 있나", "어떤 결과가 수상한가", "어떤 트레이드오프가 수용 가능한가" — 에이전트가 대신할 수 없다.
채용·팀 구성의 변화
❌ 구식 채용 방식
  • • 작은 코딩 퍼즐 풀기
  • • API 암기 여부 확인
  • • 알고리즘 화이트보드
  • • "10x 엔지니어" 기준
✅ 에이전틱 시대 채용
  • • 대규모 프로젝트를 에이전트로 구현
  • • 배포 후 공격 에이전트로 침투 시도
  • • 스펙 작성 및 diff 리뷰 능력
  • • 100x+ 엔지니어 시대
Karpathy 제안 면접: "대규모 프로젝트를 에이전트로 구현 → 배포 → 10개의 공격 에이전트가 당신의 시스템을 뚫으려 한다. 막아라." 이것이 진짜 에이전틱 엔지니어링 역량을 본다.
Agent-Native 인프라 — 다음 기회

지금 대부분의 소프트웨어는 아직 인간을 위해 설계되어 있다. 에이전트가 직접 쓸 수 있도록 재설계되어야 한다.

❌ 아직도 이렇게 생긴 것들
  • "이 URL로 이동하세요"
  • "이 버튼을 클릭하세요"
  • "설정 패널을 여세요"
  • GUI 전용 설치 과정
✅ Agent-native 인프라
  • Markdown 문서
  • CLI · API · MCP 서버
  • 구조화된 로그 · 스키마
  • 복사-붙여넣기 에이전트 지시문

Karpathy의 테스트: "build MenuGen이라고 말했을 때 에이전트가 Vercel 배포, Auth, 결제, DNS까지 자동으로 완료한다" — 그 날이 인프라가 agent-native가 된 날.

C-level 의사결정 포인트

🟢 즉시 실행

  • 개발팀 Claude Code · Codex 등 에이전틱 도구 도입 및 실험 시작
  • "10x" → "100x" 가능성으로 채용 기준·팀 규모 전략 재검토
  • 우리 제품 중 "앱이 없어야 하는 것"은 무엇인지 파악
  • 우리 서비스의 Agent-native 전환 로드맵 수립

🟡 전략적 검토

  • 가치 있고 검증 가능하지만 Labs가 미집중한 도메인 탐색 (창업·투자 기회)
  • 채용 프로세스 재설계 — 에이전틱 엔지니어링 역량 테스트
  • "이해는 위임 불가" 관점에서 교육·온보딩 재설계
  • 우리 팀의 "AI 회로" 파악 — 어디서 날고, 어디서 추락하나
